## ✂️ **청킹 (Chunking) 설정**

### **현재 설정**
```
청크 크기: 1000자
오버랩: 150자
```

**설정 위치:** `backend/app/config.py` 라인 31-32
```python
RAG_CHUNK_SIZE: int = 1000
RAG_CHUNK_OVERLAP: int = 150
```

### **청킹 전략**
```python
# backend/app/services/embedding_service.py 라인 42-48
splitter = RecursiveCharacterTextSplitter(
    chunk_size=1000,           # 1000자 단위
    chunk_overlap=150,         # 150자 중복 (컨텍스트 유지)
    separators=["\n\n", "\n", ". ", " ", ""]  # 분리 순서
)
```

### **강사님 권장값 vs 현재값**

| 문서 크기 | 강사님 권장 | 현재값 | 평가 |
|---------|-----------|-------|------|
| 작은 문서 | 500:100 | 1000:150 | ❌ 과함 |
| 중간 문서 | **1000:200** | 1000:150 | ⚠️ 오버랩 부족 |
| 큰 문서 | 1500:300 | 1000:150 | ⚠️ 너무 작음 |

**🎯 추천 변경:** `1000:150` → `1000:200` (오버랩만 +50)

## 📋 **파일 처리 흐름**

```
1️⃣ 파일 업로드/R2 동기화
   ├─ 확장자 검증 (pdf, csv, txt, docx, md)
   ├─ DB 메타 저장
   └─ 카테고리 결정 (rag vs schedule_input vs ...)

2️⃣ RAG 파일 처리 (rag, safety_manage)
   ├─ R2에서 파일 다운로드
   ├─ 텍스트 추출 (pypdf, python-docx 등)
   ├─ 청킹 (1000자 + 150자 오버랩)
   ├─ OpenAI 임베딩 호출 → 1536차원
   └─ pgvector에 저장

3️⃣ CSV 파일 처리 (schedule-data-input)
   ├─ R2에서 CSV 다운로드
   ├─ 파싱 (DictReader)
   ├─ emp_id 소문자 정규화
   └─ DB에 bulk insert

4️⃣ 검색 (챗봇)
   ├─ 사용자 질문 임베딩
   ├─ pgvector 코사인 유사도 검색
   ├─ 상위 k개 청크 반환 (기본 k=4)
   └─ LLM에 컨텍스트로 제공
```

## 🚀 **성능 고려사항**

| 항목 | 값 | 영향 |
|------|-----|------|
| 청크 크기 | 1000 | 작을수록 정확, 크면 컨텍스트 풍부 |
| 오버랩 | 150 | 클수록 안정적, 네트워크 비용 증가 |
| k값 | 4 | 크면 정보 많음, 느림 |
| 임베딩 차원 | 1536 | 표준 (변경 불가) |
